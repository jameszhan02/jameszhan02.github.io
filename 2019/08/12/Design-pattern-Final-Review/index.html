1. Bridge:

object Structural Type pattern

Intent:
Decouple an abstraction from its implementation so the two can vary independently

Class Hierachy: A superclass and all its children as a tree

Abstraction: Interface/method list

Implementation: The bodies of the methods

Motivation: was not all of child class always need all method from the super class.

Structure note:

WindowImp is an interface class have some basic method component wich both window will have.

XWindowImp and PMWindowImp they both inherit from WindowImp. just simply give implement the methods what they iherit from.

window is an abstract class with have a WindwoImp type variable in the constructor ( so it window or classes iherit it form able to access all of method in windowImp and all of class ihert form it )

Main reason use bridge in this case: more flexable, if we create only one and ihert taht super class. and we want to do the same thing we will crete some class like macIconWindow & windowIconWindow, but in this pattern we only have one IconWindow class, its just offer more flexbility to us.

  • Applicability: (when we should use this pattern)
    • want to avoid abstraction-implementation binding
    • both abstraction and implementations should be extensible ( will have more sub class)
    • You have problems with class proliferation (i.e. MacIconWindow)
    • note it should have zero differnet for the client side.
    • Abstraction forwards client requests to Implementor
  • Consequences
    • sperate implementation and interface
    • can configure or change implementation at run-time
    • Improved extensibility
    • You can extend Abstraction and Implementation separately
    • Hides implementations from clients They only see Abstraction

Useage: Cross-platform window systems, UI System

2. Decorator:

object Structural Type pattern

Intent:

dynamically to add more responsibilities to an object. Decorators provide more flexibility to subclassing for extending functionality.

Motivation:

if we only need add some extending responsibility to individual objects, not entire class. ( ex. graphical interface allow to add boders or underLine, layout pane or textbox, most likely GUI in JAVA )

Pattern structure:
have one super class ( Ishape ), all rest of classes iherit form if, concreteComponent is the leaf, but Decorator is not and it’s own tge leafs.

There are nothing special for the ConcreteComponent. and component, just basic super class and sub class relationship.

Decorator is another abstract/ ?Interface class which have a IShape object in it’s constructor, and thats how it go access the orgin object and give addtional responsibility to. which some addtional code in his Draw() method.

Decorator forwards requests to its component object.

  • decorator and its component are’t identical.

Notice: responsibilities that can be withdrawn (undo)

3.Memento

object Behavioral Type pattern

Intent:

without broke encapsulation, to capture an object intent state so can be restored to that state later (roll back ?)

Motivation:

Sometimes it’s necessary to record the internal state of an object.

required when creating checkpoints and undo mechanisms.

Pattern structure:

Memento is a closing class just hold data (object) nothing more than that. the idea of this pattern is, we dont want anyone can go modified memento except caretaker. and it’s need to have savepoint.

Originator own memento class, it’s more like a memento contorller class, with the same model as memento and have methods(action/ operation) in it can be call from caretaker and send request to, then to hanle memento.

caretaker take care about save and undo( Since Stack can do pop )

  • caretaker requests a memento from the originator, holds it for a time and passes it back to the originator.
    • notice undo method can reset nothing, if there no need for reset any data, empty stack list
    • memento only work when its created by the originator, otherwise is not doing any thing, just a class.
  • Consequences
    • mementos can be expensive. Originator may copy out lots of data
    • can use a large amount memory.
      ` - narrow and wide interface: may differect to make sure only originator can access the memento’s state
    • C# automaticly handle garbage collection no memory lick will happen.

caretaker have object for originator and a stack for memento.

4. State

object Behavioral Type pattern

Intent:

Object behavior will change if internal state changes.

in a nutshell : An object’s behavior depends on its state.

Pattern Strusture:

key point is State and it sub class. state have basic method all sub class shared. but in different sub class its have differner behavior. identify by the subclass name. but no more than nomal iherit. just structure

Having states as objects makes this much more explicit

5. Observer

object Behavioral Type pattern

Intent:

Define a one-to-many dependency between objects so that one object changes state, all its dependents are notified and updated automatically.

One mother object update, rest update too.

Pattern structure:

Subject class is have a list of Observer, it’s create when its call( constructor ).

two methods in subject will add and remove from the list.

Also, Notify will go access through the list. to tell every element go update.
(this action can be expensive)

Observer is an interface class which work as a super class.

Graph will iherit from Oberver, hold one string type and one data ( subclass of subject ) type. Notice: no one can access to the observer list except with method in subject, which can call the data since is the subclass.

and once data setData call it’s go forward request Notify(). and notify loop through the all list ( the point of this list is to see how many ‘subelement’ we have want to look up to our object ). then will call update() method for each element. to show some action.

  • Subject
    • Knows its observers (has a list of Observer)
    • Provides an interface for attaching and detaching Observers ( subscribe & unsubscribe )
  • Observer
    • Defines an updating interface for objects that should be notified of changes in a subject ( All of subject in the list will be call with notify and do thier action )

6. Facade

object Structural Type pattern

Intent:

You want to provide a simple interface to a complex subsystem.

Facade is a overall interface include all of subclasses object in one. so in program (main method) we dont have to create all of class to object for using, but just create one facade object. make code clean.

there are nothing more fanc than that.

  • cons:
    • reducing the number of objects clients deal with
    • Client only knows about single (1) Facade class - not all 5 compiler classes
    • Does not prevent clients from using subsystem classes if they need to ( even we have facade class to contain all of subclass, we still able to call each subcalss object indivual )

7.Mediator

object Behavioral Type pattern

mediator its more like orgnize connections between different object. more objects more complex to design

it’s why have a big benfit to use Mediator pattern

  • Good Consequences

    • all of object now are Encapsulation boundaries
    • Limits subclassing
    • Simplifies object protocols
      • Many:many interactions become one:many - easier to maintain
    • bad Consequences
      • Mediator can be compleax to design the relationship between classes, will be hard to maintain

Be use for UI a lot
